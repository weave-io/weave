## Overview

`@weaveio/weave-adapter-claude-code` translates Weave config into a Claude Code plugin directory. It uses a **two-plugin model**:

1. **Bootstrap plugin** (`weave-bootstrap-plugin/`) — static, version-controlled, installed once. Its `SessionStart` hook runs `bun run weave compose --adapter claude-code` automatically every time Claude Code opens a session in the project, regenerating the generated plugin from the current `.weave/config.weave`.

2. **Generated plugin** (`.weave/plugins/claude-code/`) — produced by the compose command, gitignored, regenerated on every session start. Contains materialized agent markdown files, copied skill files, and a `settings.json` declaring the primary agent.

Unlike the OpenCode adapter (which makes SDK API calls at runtime), the Claude Code adapter is **pure file I/O** — no runtime client is required. Every agent is a `.md` file with YAML frontmatter written to the generated plugin directory.

## Architecture

```
weave-bootstrap-plugin/              ← static, version-controlled, installed once
├── .claude-plugin/plugin.json       ← name: "weave-bootstrap", version: "1.0.0"
├── hooks/hooks.json                 ← SessionStart → bun run weave compose --adapter claude-code
└── skills/compose/SKILL.md         ← /weave:compose for manual regeneration

.weave/plugins/claude-code/          ← generated, gitignored, regenerated every session
├── .claude-plugin/plugin.json       ← name: "weave", version: "1.0.0"
├── agents/
│   ├── loom.md                      ← primary orchestrator agent
│   ├── shuttle.md                   ← base domain specialist
│   ├── shuttle-<category>.md        ← one per declared category
│   └── ...
├── skills/<skill-name>/SKILL.md     ← copied from resolved skill sources
└── settings.json                    ← { "agent": "loom" }
```

Each agent file uses YAML frontmatter followed by the composed prompt body:

```markdown
---
name: shuttle-backend
description: "Shuttle (Domain Specialist) — Backend: APIs, services, persistence"
model: sonnet
tools:
  - Read
  - Glob
  - Grep
  - Write
  - Edit
  - Bash
disallowedTools:
  - Agent
maxTurns: 50
---

<composed prompt content>
```

## Installation

### Prerequisites

Install the Weave CLI (which includes the Claude Code adapter):

```bash
bun add -D @weaveio/weave-cli
```

Or for preview versions:

```bash
bun add -D @weaveio/weave-cli@preview
```

### 1. Initialize the project

From your project root (where `.weave/config.weave` lives):

```bash
bun run weave compose --adapter claude-code --init
```

This does two things:
- Copies the **bootstrap plugin** to `./weave-bootstrap-plugin/`
- Runs the first composition, generating `.weave/plugins/claude-code/` with all agents

### 2. Gitignore the generated plugin

Add to `.gitignore`:

```
.weave/plugins/
```

The `weave-bootstrap-plugin/` directory **should** be committed — it's static and shared with the team.

### 3. Launch Claude Code with both plugins

```bash
claude --plugin-dir ./weave-bootstrap-plugin --plugin-dir .weave/plugins/claude-code
```

Both `--plugin-dir` flags are required:
- `./weave-bootstrap-plugin` — provides the `SessionStart` hook and `/weave:compose` skill
- `.weave/plugins/claude-code` — provides the materialized agents and settings (generated by the hook)

**Tip:** Add this as a script in your `package.json` for convenience:

```json
{
  "scripts": {
    "claude": "claude --plugin-dir ./weave-bootstrap-plugin --plugin-dir .weave/plugins/claude-code"
  }
}
```

## Regeneration

### Automatic — SessionStart hook

Every time Claude Code opens a session with the bootstrap plugin loaded, it runs:

```bash
bun run weave compose --adapter claude-code
```

This reads `.weave/config.weave`, materializes all agents, and writes the generated plugin directory. If `.weave/config.weave` does not exist, the command exits cleanly with a warning and the session opens normally — the hook will not block a non-Weave project.

### Manual — `/weave:compose`

To regenerate mid-session without restarting:

1. Run `/weave:compose` in Claude Code.
2. After the command completes, run `/reload-plugins` to apply the updated files in the current session.

## Tool Policy Mapping

The adapter translates Weave's abstract `tool_policy` capabilities into Claude Code's concrete `tools` (allowed) and `disallowedTools` (blocked) frontmatter arrays.

| Weave capability | `allow` or `ask` → `tools` | `deny` → `disallowedTools` |
|---|---|---|
| `read` | `Read`, `Glob`, `Grep` | `Read`, `Glob`, `Grep` |
| `write` | `Write`, `Edit` | `Write`, `Edit` |
| `execute` | `Bash` | `Bash` |
| `delegate` | `Agent`, `Task` | `Agent`, `Task` |
| `network` | _(no mapping)_ | _(no mapping)_ |

**`ask` behaviour**: `ask` maps to `tools` (same as `allow`). Claude Code's native permission prompting handles the user-confirmation flow — the adapter does not set `permissionMode` on any generated agent (Claude Code restricts this field at the plugin level; setting it produces a security error).

**`network`**: There is no stable Claude Code tool name for network access, so `network` policy is intentionally omitted from both arrays regardless of its value.

---

## Model Alias Mapping

Weave model preference strings are mapped to Claude Code model aliases. The first entry in the agent's `models` array is used; if no match is found, the string is forwarded verbatim as the model field value.

| Weave model string | Claude Code `model` field |
|---|---|
| `claude-sonnet-4-5`, `claude-sonnet-4`, `claude-sonnet` | `sonnet` |
| `claude-opus-4-5`, `claude-opus-4`, `claude-opus` | `opus` |
| `claude-haiku-3-5`, `claude-haiku-3`, `claude-haiku` | `haiku` |
| Any other string | Pass through verbatim |

If the `models` array is empty, the `model` field is omitted from the frontmatter and Claude Code uses its default model.

## Differences from OpenCode Adapter

| Aspect | OpenCode adapter | Claude Code adapter |
|---|---|---|
| Materialization | SDK API calls at runtime | File generation (plugin directory) |
| Client dependency | `OpenCodeClientFacade` | None — pure file I/O |
| Reconciliation | List → create/update via SDK | Overwrite files on every regeneration |
| Refresh mechanism | SDK re-registers on change | `/reload-plugins` or session restart |
| Tool policy | `permissionMode` + patches | `tools` / `disallowedTools` arrays |
| Model format | Direct model IDs | Aliases (`sonnet`, `opus`, `haiku`) or full IDs |
| Runtime lifecycle | Plugin hooks inside OpenCode | `SessionStart` hook via bootstrap plugin |

## Edge Cases

### First run — manual `/reload-plugins` required

On the very first session open, the `SessionStart` hook writes the generated plugin to disk, but Claude Code has already loaded plugins for that session. The generated plugin will not be active until the next session start or until you manually run `/reload-plugins` after the initial compose completes.

**Workaround**: After the first `claude --plugin-dir ...` launch, run `/reload-plugins` once to activate the generated plugin immediately.

### No `.weave/config.weave` — graceful exit

If `.weave/config.weave` does not exist when the `SessionStart` hook fires, the compose command exits `0` with a structured log warning. The session opens normally. This protects non-Weave projects that happen to have the bootstrap plugin loaded.

### Mid-session config changes

If you edit `.weave/config.weave` during a session:

1. Run `/weave:compose` to regenerate the plugin directory with the new config.
2. Run `/reload-plugins` to load the updated agents into the current session.

Agents that were already active in the current conversation are not replaced retroactively — only new sub-agent invocations after `/reload-plugins` will use the updated definitions.

### `loom` agent not present

If no agent named `loom` is declared in your config, the adapter logs a `NoLoomAgentError` warning and omits `settings.json` from the generated plugin. Claude Code will use its default agent selection. Add a `loom` agent to your `.weave/config.weave` to enable the primary-agent declaration.
