## Summary

Task 4 extends `convertLegacyJsonc()` in `packages/cli/src/commands/init.ts` with semantic mapping for the five remaining legacy JSONC top-level fields: `agents`, `custom_agents`, `categories`, and the per-entry sub-fields `model`/`fallback_models`, `tools`, and `prompt_file`.

All conversion follows the best-effort partial-success model established in Task 3: supported content is written even when warnings are emitted for skipped fields. The Task 2 validation gate (`parseConfig()` before any file write) and Task 3 warning semantics are preserved intact.

**Namespace semantics (corrected in retry)**: The `agents` top-level key is strictly for overriding existing builtin agent names. Non-builtin names under `agents` are warned and skipped — they do not silently become new agents. New agents must come from `custom_agents` instead. This distinction is enforced by checking `BUILTIN_AGENT_NAMES.has(agentName)` before processing each `agents` entry.

## Implementation Context

### New constants added to `init.ts`

| Constant | Purpose |
|---|---|
| `BUILTIN_AGENT_NAMES` | Set of 8 builtin names (`loom`, `tapestry`, `shuttle`, `pattern`, `thread`, `spindle`, `weft`, `warp`) used for collision detection in `custom_agents` |
| `LEGACY_TOOL_TO_CAPABILITY` | Map from clearly known legacy tool names (`write`, `edit`, `bash`, `task`, `read`, `web_search`, `web_fetch`) to abstract capability buckets |
| `AMBIGUOUS_LEGACY_TOOLS` | Set of harness-specific tool names (`call_weave_agent`, `todowrite`, `mcp`, `computer`) that are warned and skipped |

### New helper functions added to `init.ts`

| Function | Subtask | Behavior |
|---|---|---|
| `isPromptFileSafe(path)` | 4.7/4.8 | Returns `true` only for bare filenames with no directory separators |
| `convertLegacyTools(tools, label)` | 4.6 | Maps known tool names to `tool_policy` lines; warns on ambiguous/unknown |
| `convertLegacyModels(entry, label)` | 4.4 | Builds `models [primary, ...fallbacks]` line |
| `convertLegacyPromptFile(value, label)` | 4.7/4.8 | Returns `prompt_file "..."` line for safe paths; warns and skips unsafe |
| `convertLegacyAgentEntry(name, entry, warnings)` | 4.1 | Converts a builtin agent override entry into an `agent <name> { ... }` block |
| `convertLegacyCustomAgent(name, entry, warnings)` | 4.2 | Converts a non-colliding custom agent into an `agent <name> { ... }` block |
| `convertLegacyCategory(name, entry, warnings)` | 4.5 | Converts a category entry into a `category <name> { ... }` block |

### New conversion cases in `convertLegacyJsonc()`

Three new top-level key handlers were added before the "Unknown field" catch-all:

- `agents` → calls `convertLegacyAgentEntry()` for each entry (4.1)
- `custom_agents` → collision-checks against `BUILTIN_AGENT_NAMES`, then calls `convertLegacyCustomAgent()` for non-colliding entries (4.2, 4.3)
- `categories` → calls `convertLegacyCategory()` for each entry (4.5)

Warning output (sanitized):

```
⚠  Migration warnings — the following legacy fields were skipped:

  • custom_agents.loom: "loom" collides with a builtin agent name; skipped to avoid silently overriding the builtin
  • agents.shuttle.tools.call_weave_agent: "call_weave_agent" is a harness-specific tool name that cannot be mapped to an abstract tool_policy capability; skipped
  • agents.shuttle.prompt_file: "subdir/shuttle.md" contains directory components and cannot be safely translated to the current .weave/prompts/ convention; skipped
```

Key properties:
- No source file content is dumped in any warning message
- Each warning names the exact field path and gives an explicit reason
- Migration exits with code `0` despite warnings
- Destination file is written with the non-skipped content

## Design Decisions

### 4.1 — `agents` as builtin overrides, not new agents

Legacy `agents` entries are treated as overrides of existing builtin names. The converter generates `agent <name> { ... }` blocks with only the overridden fields. This is correct because the current DSL merges user-declared agent blocks with builtin defaults — declaring `agent loom { temperature 0.2 }` overrides only temperature, leaving all other builtin loom config intact.

### 4.3 — Collision detection for `custom_agents`

`custom_agents` entries whose names match any of the 8 builtin names are warned and skipped. This prevents silent builtin overrides that would change builtin behavior without the user explicitly choosing to do so via the `agents` key. The warning message names the exact field and explains the reason.

### 4.5 — Categories as `category` blocks, not flattened shuttle agents

The legacy system generated `shuttle-{category}` agents by flattening category config into standalone agent entries. The current DSL generates `shuttle-{category}` semantics automatically from `category <name>` blocks. The converter uses `category` blocks exclusively — no `agent shuttle-backend` entries are generated.

### 4.6 — Conservative tool mapping

Only tool names with a clear, unambiguous mapping to abstract capability buckets are converted. Harness-specific tools (`call_weave_agent`, `todowrite`, `mcp`, `computer`) are warned and skipped because they have no abstract equivalent. Unknown tool names are also warned and skipped. This is intentionally conservative: it is better to warn and let the user add the correct `tool_policy` manually than to silently map to the wrong capability.

### 4.7/4.8 — Prompt file safety

Only bare filenames (no directory separators, no `..` prefix) are preserved as `prompt_file` references. Paths with directory components cannot be safely translated to the current `.weave/prompts/` convention because the legacy system may have used arbitrary filesystem paths. Warning messages do not dump the source file content.
