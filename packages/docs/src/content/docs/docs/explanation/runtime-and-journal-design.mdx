---
title: Runtime and Journal Design
description: Why Weave has an engine-owned runtime store and sanitized journal.
---

The runtime store is a narrow engine-owned persistence boundary. It records Weave execution concepts such as workflow instances, leases, artifacts, session observations, and journal entries.

This state belongs to Weave, not to OpenCode or any future harness, because lifecycle semantics are shared across adapters. A single runtime vocabulary lets command operations inspect, resume, pause, and audit execution without depending on harness-private storage.

The journal is intentionally sanitized. It should help operators understand state transitions without exposing raw prompts, completions, transcripts, provider payloads, cookies, tokens, or authorization headers.

Adapters may contribute safe observations, but they should not treat the runtime database as their private store.
