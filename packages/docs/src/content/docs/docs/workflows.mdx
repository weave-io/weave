<p class="lede" style="margin-top:var(--s-6)">A workflow is a directed acyclic graph of agent roles. You declare its stages; Weave infers their dependencies from references, rejects cycles and dead-ends at compile time, and emits a typed graph the runtime contract guarantees.</p>

## Overview

<p>A <code>workflow</code> block names an ordered set of stages. Declaration order is for reading; the actual execution order is derived from how stages reference one another. Because the result is a DAG, Weave can verify the whole flow before any harness runs a single token.</p>

<p>Every stage resolves to a role — by name, or through a <a class="link" href={`${base}docs/`}>category</a> that routes by capability. A workflow that references an undefined role, forms a cycle, or strands a stage with no path to a terminal will not compile.</p>

## Syntax

<p>The smallest useful workflow is a linear chain. Stages name the role and the skill they invoke.</p>

<CodeBlock name="analyze.weave" id="c-analyze" code={cAnalyze} />

<Callout label="Note">
  <p>The <code>stages</code> list is <strong style="color:var(--text)">declaration order</strong>, not execution order. <code>report</code> runs after <code>survey</code> only because it references <code>survey</code> in <code>from</code>. Reorder the list freely — the compiled graph is unchanged.</p>
</Callout>

### Stage shape

<p>A stage binds a name to a role, a skill, and arguments:</p>

<div class="term"><span class="dim"># stage: role.skill(arg: value)</span><br /><span class="p">›</span> survey: explorer.scan(depth: "shallow")</div>

<p>Stage names match <code>[a-z][a-z0-9_]&#123;0,31&#125;</code> — lowercase, digits and underscores, up to 32 characters. Names are unique within a workflow and become the node identifiers in the compiled graph.</p>

## Inputs & dependencies

<p>Dependencies are edges, and edges are inferred. When a stage names another stage in its arguments, Weave records a dependency and orders the graph accordingly. Nothing is implicit beyond that — there are no hidden ordering rules.</p>

<CodeBlock name="migrate_repo.weave" id="c-migrate" code={cMigrate} />

<Callout label="Spec" variant="spec">
  <p>The compiler <strong style="color:var(--text)">MUST</strong> reject any workflow that contains a cycle (<code>E2101</code>) or references an undefined dependency (<code>E2102</code>). Both are detected during normalization, before adapters are invoked.</p>
</Callout>

### Fan-in with +

<p>The <code>+</code> operator fans multiple roles into one stage. In <code>migrate_repo</code>, <code>verify</code> binds both <code>reviewer</code> and <code>security</code> to a single audit node — both must complete before the stage resolves. Fan-in is the only place two roles share a stage; everywhere else, one stage is one role.</p>

## The compiled graph

<p>Normalization turns the source above into a typed graph. Linear references become a chain; <code>parallel: true</code> marks a node as concurrently executable; the <code>+</code> fan-in expands into a join.</p>

<WorkflowGraph />

## Parallelism

<p>A stage marked <code>parallel: true</code> may fan its work across multiple concurrent invocations of its role. The runtime decides the degree of concurrency; the graph only declares that the node is safe to parallelize. Downstream stages still wait for the parallel node to fully resolve before they begin.</p>

<Callout label="Warning" variant="warn">
  <p>Parallel stages <strong style="color:var(--text)">MUST</strong> be side-effect-isolated. If two concurrent invocations write the same path, the runtime contract does not define a merge order — partition the work, or leave the stage sequential.</p>
</Callout>

## Termination

<p>Every workflow needs at least one terminal stage — a node with no outgoing edges that the graph can reach from its entry. A stage that no terminal depends on is unreachable and is reported as a warning; a graph with no terminal at all is a hard error.</p>

<div class="term"><span class="p">$</span> weave build broken.weave<br /><span class="err">✗ E2103  unreachable terminal: stage 'cleanup' has no path to a terminal node</span></div>

## Runtime contract

<p>Adapters consume the compiled graph through a small, stable surface. Core defines the contract; adapters implement it without mutating intent.</p>

<div class="deflist">
  <div class="row"><dt>stage.run</dt><dd>The harness <span class="req">MUST</span> invoke a stage only after all its dependencies have resolved.</dd></div>
  <div class="row"><dt>stage.parallel</dt><dd>The harness <span class="req">MAY</span> parallelize a node marked <code>parallel: true</code>; it <span class="req">MUST NOT</span> parallelize any other.</dd></div>
  <div class="row"><dt>stage.timeout</dt><dd>The harness <span class="req">SHOULD</span> honor a per-stage timeout and surface expiry as a stage failure.</dd></div>
  <div class="row"><dt>workflow.cancel()</dt><dd>Cancellation <span class="req">MUST</span> halt unstarted stages and <span class="req">SHOULD</span> signal in-flight ones.</dd></div>
  <div class="row"><dt>events</dt><dd>The harness <span class="req">SHOULD</span> emit <code>stage.start</code>, <code>stage.end</code> and <code>stage.error</code> for observability.</dd></div>
</div>

## Common errors

<table class="spec-table">
  <thead><tr><th>Code</th><th>Meaning</th><th>Resolution</th></tr></thead>
  <tbody>
    <tr><td class="code-col">E2101</td><td>Cycle detected in stage graph</td><td>Remove the back-reference; a workflow must be acyclic.</td></tr>
    <tr><td class="code-col">E2102</td><td>Undefined dependency</td><td>A stage references a name that is not declared. Fix the reference or add the stage.</td></tr>
    <tr><td class="code-col">E2103</td><td>Unreachable terminal</td><td>A stage has no path to a terminal node. Connect it or remove it.</td></tr>
    <tr><td class="code-col">E2104</td><td>Role missing skill</td><td>The bound role does not declare the invoked skill. Add the skill or change roles.</td></tr>
    <tr><td class="code-col">W2110</td><td>Unused stage</td><td>Stage is declared but never referenced. Warning only — safe to prune.</td></tr>
  </tbody>
</table>
