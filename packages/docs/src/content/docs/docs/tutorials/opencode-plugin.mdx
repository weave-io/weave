This tutorial connects a valid `.weave/config.weave` to OpenCode, the only implemented adapter today.

## Prerequisites

- You completed the [Quickstart](../quickstart/).
- OpenCode is installed and has a project config file.
- Workspace dependencies are installed with Bun.

## Initialize for OpenCode

Run:

```bash
bun packages/cli/src/main.ts init --scope local --harness opencode --yes
```

The initializer detects harnesses and installs only supported harness installers. Unsupported requested harnesses fail with a clear message.

## Add the plugin entry manually if needed

OpenCode must load the plugin subpath export, not the bare package:

```jsonc
{
  "plugin": ["@weave/adapter-opencode/plugin"]
}
```

The adapter code documents this as the supported entry point because the bare package exports non-plugin values.

## Validate Weave config

```bash
bun packages/cli/src/main.ts validate --project
```

Fix parse or validation errors before starting OpenCode.

## Start OpenCode

Run OpenCode with its own command:

```bash
opencode
```

Weave does not start OpenCode for you. `weave run` intentionally exits with guidance to use the harness runtime command.

## Verify config materialization

Use OpenCode's config debugging surface if available:

```bash
opencode debug config
```

The plugin config hook injects translated Weave agents into OpenCode config and registers prompt-based commands including `/weave:start` and `/start-work`.

## What is implemented today

- Agent descriptor materialization for OpenCode.
- Abstract tool policy mapping into OpenCode config.
- Model resolution with adapter-owned model context.
- Prompt-based `/weave:start` and `/start-work` command templates.
- Runtime command projection helpers for `/weave:run`, `/weave:status`, `/weave:abort`, `/weave:advance`, and `/weave:health` in adapter code.
